# Plan de Mejoras Detalladas — `mapj` CLI

## 1. Calidad del Código Go

### 1.1 Convenciones y Nomenclatura
- **`pkg/confluence/client.go:57`** — Método `_getHeaders()` con guion bajo no es idiomático Go. Renombrar a `defaultHeaders()` o convertir en variable privada `var defaultHeaders = map[string]string{...}`.
- **`pkg/confluence/client.go:13-23`** — El struct `Client` tiene campos redundantes: `Token`, `Username`, `Password`, `UseBasic`, `basicUser`, `basicPass`. Debería simplificarse a un solo campo `authScheme` + `credentials`.
- **`internal/cli/protheus.go:88-91`** — Variables de paquete sin agrupar en un solo `var()` block. Inconsistente con el resto del código.

### 1.2 Manejo de Errores Ignorados
- **`pkg/confluence/export.go:100`** — `_ = WriteManifest(...)` ignora errores silenciosamente. Si el manifest falla, el usuario no se entera.
- **`pkg/confluence/export.go:265`** — `_ = WriteManifest(...)` mismo problema.
- **`pkg/confluence/logger.go:96-97`** — `l.errorFile.Write(...)` ignora el error retornado. Si el disco se llena, los errores se pierden silenciosamente.
- **`internal/cli/root.go:58-60`** — `PersistentPreRunE` retorna `nil` siempre (hook vacío). Eliminar o usar para validación global.

### 1.3 Ineficiencias
- **`pkg/protheus/query.go:77-81`** — `sql.Open()` + `db.Close()` por cada query. `sql.Open` no abre la conexión realmente pero el patrón es subóptimo. Debería haber un `Client` con `*sql.DB` como campo reutilizable.
- **`pkg/protheus/query.go:134-138`** — Mismo problema en `Ping()`: abre/cierra conexión nueva cada vez.
- **`pkg/confluence/export.go:149`** — Dentro del loop de `ExportPages`, se hace `c.GetPage(ctx, pageID, "space,metadata.labels,ancestors")` para cada página exitosa, duplicando la llamada API (ya se obtuvo la página en `exportSinglePage`). Debería reutilizar el `Page` ya obtenido.
- **`pkg/confluence/markdown.go:21-41`** — `ConvertToMarkdown` crea un nuevo `converter.NewConverter()` con plugins en cada llamada. Debería ser un singleton/pool para evitar re-inicialización costosa en exportaciones masivas.

### 1.4 Duplicación de Código
- **`pkg/confluence/export.go`** — La construcción de `ManifestEntry` se repite 3 veces (líneas ~90-99, ~150-168, ~247-264). Extraer a función `newManifestEntry(page, exportPath, baseURL)`.
- **`internal/cli/confluence.go:146-190`** y **`internal/cli/tdn.go:337-359`** — `getConfluenceClient()` y `getTDNClient()` comparten lógica de carga de credenciales. Unificar en factory package.
- **`internal/cli/protheus.go:230-238`** y **`internal/cli/protheus_connection.go:484-495`** — `protheusVPNHint()` y `vpnHintForServer()` son esencialmente la misma función con firmas distintas.

### 1.5 Seguridad de Tipos
- **`pkg/confluence/search.go:213`** — `Total: raw.Size` es incorrecto. El comentario dice "REST API doesn't return total easily" pero se asigna `Size` como `Total`, lo que confunde al agente sobre cuántos resultados totales existen.
- **`internal/cli/confluence.go:313,323`** — Emojis en `fmt.Fprintf(os.Stderr, ...)` inconsistentes con el resto del código y pueden causar problemas en terminales sin soporte UTF-8.

## 2. Arquitectura

### 2.1 Falta de Interfaces/Abstracciones
- No hay interfaz para `confluence.Client` ni `protheus.Client`. Esto imposibilita mockear en tests de `internal/cli/`.
- **Solución**: Definir `type ConfluenceAPI interface { Search(...) ...; Export(...) ...; ... }` y `type ProtheusAPI interface { Query(...) ...; Ping(...) ... }`.

### 2.2 Acoplamiento CLI↔pkg
- **`internal/cli/tdn.go:210-258`** — `runSearchExportPipeline()` debería vivir en `pkg/confluence/` o en un paquete `internal/pipeline/`. El CLI no debería contener lógica de orquestación de negocio.
- **`internal/cli/confluence.go:146-190`** — `getConfluenceClient()` conoce sobre `auth.Store`, `output.Formatter`, y construye errores. Debería ser un factory puro.

### 2.3 Paquete `internal/auth`
- **`store.go:161`** — `SetKey(key string)` debería recibir `[]byte` para consistencia con `GetEncryptionKey()`.
- **`store.go:113-123`** — `GetEncryptionKey()` expone la lógica de derivación. Debería ser interna y solo exportar `NewStore()`.
- **`login.go:227-238`** — `loginJSON()` y `outputFlagFromArgs()` crean dependencia circular evasiva (parsear `os.Args` manualmente). Mejor solución: inyectar formatter desde CLI.

### 2.4 Manejo de Estado Global
- Todos los flags de CLI son variables de paquete (`var tdnSpace string`, `var confluenceOutputPath string`, etc.). Esto causa problemas en tests concurrentes y re-entrada.
- **Solución**: Usar `cmd.Context()` o structs de configuración por comando.

## 4. Seguridad

### 4.1 SQL Injection / Validación
- **`pkg/protheus/query.go:45-70`** — Validación por regex es evadible:
  - `/* comment */ INSERT INTO ...` — el regex de comentarios no maneja anidación
  - Unicode normalization puede evadir `\b` boundaries
  - `SEL ECT` con caracteres invisibles (zero-width space)
  - **Solución real**: Usar parser SQL como `github.com/xwb1989/sqlparser` o `github.com/pganalyze/pg_query_go` (adaptado para T-SQL).

### 4.2 Cifrado de Credenciales
- **`pkg/protheus/query.go:42`** — `encrypt=disable` hardcodeado en connection string. Las credenciales de SQL Server viajan en texto plano por la red.
  - **Solución**: `--encrypt` flag con opciones `disable`, `false`, `true`, `strict`.
- **`internal/auth/store.go`** — AES-256-GCM con key derivada de hostname+user+homeDir es débil:
  - Si alguien conoce hostname y username, puede derivar la key
  - No hay salt ni iterations (KDF trivial)
  - **Solución**: Usar `golang.org/x/crypto/argon2` o `scrypt` con salt aleatorio almacenado junto al ciphertext.

### 4.3 Exposición de Secretos
- **`internal/cli/protheus_connection.go:156-157`** — `connListRun` muestra `User` en plaintext en el output. Debería enmascararse como la password.
- **`internal/cli/protheus_connection.go:350-358`** — `connShowRun` enmascara password pero muestra user completo. Aceptable, pero documentar.

### 4.4 Rate Limiting
- No hay rate limiting client-side para evitar 429s en exportaciones masivas.
- **Solución**: `time.Sleep()` configurable entre requests o token bucket con `golang.org/x/time/rate`.

## 5. Diseño CLI para Agentes

### 5.1 Flags Faltantes
- **`--timeout`** — No configurable. Hardcodeado a 30s en HTTP client y 15s en ping.
- **`--dry-run`** — Para exportaciones recursivas, el agente no puede predecir cuántos archivos se crearán sin ejecutar.
- **`--version`** — No existe `mapj --version`. Crítico para debugging de agentes.
- **`--max-retries`** — Para operaciones retryables, el agente no sabe cuántas veces reintentar automáticamente.
- **`--proxy`** — Sin soporte para proxy HTTP (común en entornos corporativos con VPN).

### 5.2 Output
- **CSV no escapa comas correctamente** — Documentado como known gotcha (#6 en CONTRIBUTING.md). `buildCSVPayload` en `protheus.go:214-227` usa `fmt.Sprintf("%v", f)` sin quoting RFC 4180 apropiado. El `csvRow` en `formatter.go:84-98` sí hace quoting correcto, pero nunca se usa porque `buildCSVPayload` construye `CSVPayload` con strings ya formateados.
- **No hay `--output jsonl`** — Para streams grandes, JSONL es más eficiente que JSON array.
- **El output de error mezcla `\n` en el mensaje** — `protheus.go:156` concatena `msg + "\n" + hint`, lo que rompe el JSON compacto en una línea.

### 5.3 Exit Codes
- **`internal/errors/codes.go:28-43`** — `MapErrorToCode` usa string matching (`strings.Contains(errStr, "USAGE_ERROR")`). Frágil: si el mensaje de error cambia, el mapeo se rompe.
- **Solución**: Definir tipos de error custom: `type UsageError struct{...}` con `Code() int`.

## 7. Manejo de Errores

### 7.1 Tipos de Error
- **Todo el CLI** usa `errors.New("USAGE_ERROR")` como valor de retorno para que `MapErrorToCode` lo detecte por string matching. Esto es anti-patrón.
- **Solución**:
  ```go
  type ExitCoder interface { ExitCode() int }
  type UsageError struct { Code, Message, Hint string }
  func (e UsageError) Error() string { return e.Message }
  func (e UsageError) ExitCode() int { return errors.ExitUsage }
  ```

### 7.2 Retry Automático
- **`pkg/confluence/client.go`** — No hay retry automático para errores 429/503. El agente debe reintentar manualmente.
- **Solución**: Retry con exponential backoff en `do()`:
  ```go
  // 3 retries: 1s, 2s, 4s
  for attempt := 0; attempt < 4; attempt++ {
      resp, err := c.httpClient.Do(req)
      if resp.StatusCode == 429 {
          time.Sleep(time.Second << attempt)
          continue
      }
      // ...
  }
  ```

### 7.3 Panic Recovery
- **`pkg/confluence/markdown.go:35`** — Si `conv.ConvertString` retorna error, se retorna string vacío silenciosamente. Debería loguear el error.
- No hay `recover()` en ningún punto del CLI. Un panic en `html-to-markdown` crashearía toda la CLI.

## 8. Performance

### 8.1 Exportaciones Masivas
- **`pkg/confluence/export.go:135-173`** — `ExportPages` es secuencial (un page a la vez). Para 342 páginas, esto es lento.
- **Solución**: Worker pool con semáforo (como `enrichWithChildCount`):
  ```go
  sem := make(chan struct{}, 5) // 5 concurrent exports
  ```

### 8.2 Allocaciones
- **`pkg/confluence/markdown.go`** — `cleanupMarkdown` usa regex que compila en cada llamada. Mover a variable package-level.
- **`pkg/confluence/search.go:164`** — `make([]PageRef, 0, len(raw.Results))` debería usar capacidad conocida si es posible.

### 8.3 Memoria
- **`pkg/confluence/export.go:149`** — `GetPage` con expand completo carga todo el HTML en memoria. Para páginas grandes (>1MB), esto es problemático en exportaciones concurrentes.
- **Solución**: Streaming o límite de tamaño con fallback.
